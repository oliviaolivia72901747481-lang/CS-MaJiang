# v0.8.9-state-invariant-audit 报告

## 1. 问题背景

v0.8.8 后 UI 与移动端去重工作已完成，本轮转入牌局状态一致性审计。目标不是重构规则引擎，而是补上可复用的只读不变量检查工具，并用少量高风险流程测试覆盖吃、碰、明杠、暗杠、补杠、杠后补牌与杠后出牌回落。

## 2. 审计目标

- 建立结构化 violations 的状态不变量审计入口。
- 验证真实 `GameState` 中手牌、副露、弃牌河、牌墙、`lastDiscard`、`pendingActions` 与回合阶段的一致性。
- 验证 `PlayerVisibleView` 不泄露对手暗手与完整暗杠牌面。
- 若发现真实状态或可见视图 bug，仅做最小修复。

## 3. 当前状态结构审查结果

`GameState` 的主要容器为：

- `players[].hand`
- `players[].melds`
- `players[].discards`
- `state.discards`
- `state.wall`
- `state.lastDiscard`
- `state.pendingActions`
- `state.activeSeats`

弃牌河存在两份镜像：`state.discards[seat]` 是服务端全局弃牌河，`player.discards` 是玩家对象镜像。审计工具以 `state.discards` 计入牌总量，并额外检查两者 instanceId 顺序一致，避免重复计数。

## 4. 初始牌总数来源

初始牌总数来自 `createChangshaTiles()`，当前牌集为万、筒、条 1-9，每种 4 张，总计 108 张。审计工具不硬编码 108，而是读取生成器结果作为总量与单张 tileKey 数量上限来源。

## 5. 状态中所有牌容器说明

- 手牌：`players[].hand`
- 副露：`players[].melds[].tiles`
- 弃牌河：`state.discards[seat]`
- 玩家弃牌镜像：`players[].discards`
- 牌墙：`state.wall`
- 最新弃牌指针：`state.lastDiscard` 只作为指针，不重复计入总量
- 抢杠胡时的 `lastDiscard` 指针允许指向补杠牌，但不计入弃牌河

## 6. 玩家持牌数量公式

审计采用物理持牌量：

```text
玩家物理持牌量 = hand.length + sum(meld.tiles.length)
```

杠计数：

```text
kongCount = mingGang + anGang + buGang 的副露组数
```

基本公式：

- 当前等待出牌玩家：`14 + kongCount`
- 非当前等待出牌玩家：`13 + kongCount`
- 吃、碰、明杠响应后的出牌玩家会成为 `currentSeat`
- 明杠、暗杠、补杠进入 `gangReplacement`，补牌后回到 `playing` 并保持 `14 + kongCount`
- 杠后打出一张牌后回落到 `13 + kongCount`

`waitingForResponses` 有两类情况：普通弃牌响应时所有玩家按 `13 + kongCount`；自摸/暗杠/补杠等当前玩家自响应时当前玩家按 `14 + kongCount`。

- 出牌后 `getPlayerPhysicalTileCount(player) === 13 + kongCount`
- 弃牌河增加 1
- 状态仍通过总量与副露审计

## 13. 暗牌隔离复核结果

发现并修复一个真实可见视图问题：对手视角的 `opponent.melds` 原样返回 `anGang` 的 4 张真实 tile，JSON 层面会泄露完整暗杠牌面。

最小修复：

- `server-visible-view.ts` 对对手 `anGang` 且 `exposed=false` 的副露返回 4 张带 `hidden: true` 的占位 tile。
- 自己视角 `self.melds` 不变，仍可看到完整暗杠。
- 桌面与移动端 meld 渲染兼容 `tile.hidden === true`，不展示占位牌面。

## 19. 发现的问题与修复说明

发现问题：

- 对手可见视图中 `anGang` 副露原样携带 4 张真实 tile，存在完整暗杠牌面泄露风险。

修复：

- 仅在构建对手视角时掩码 `anGang` 且 `exposed=false` 的副露 tile。
- 自己视角与结算以外的对手手牌隐藏逻辑保持不变。
- UI 仅增加对 `tile.hidden` 的兼容渲染，不调整布局。

未发现需要重构规则引擎的问题。

## 20. 已知限制

- 审计工具当前是只读工具，未接入生产时实时弹错。
- 牌墙“每次动作减少 1”的历史差分审计依赖流程测试覆盖；工具本身审计当前状态快照。
- `expectedPlayerTileCount` 对公开 API 保留简单公式；`auditPlayerTileCounts` 内部针对 `waitingForResponses` 做了阶段细分。
- 抢杠胡场景允许 `lastDiscard` 指向补杠牌但不在弃牌河中，此分支做了兼容，未在本轮新增独立抢杠胡专项测试。

## 21. 下一阶段建议

- 将 `auditGameStateInvariants` 接入开发模式诊断面板或服务端 debug 日志，但不要在生产路径强制运行。
- 后续增加抢杠胡、海底、多人 inactive seat 的专项 invariant tests。
- 若继续做网络层稳定性，可在 `submitPlayerAction` 完成后仅开发模式采样调用审计工具，便于尽早定位状态异常。
