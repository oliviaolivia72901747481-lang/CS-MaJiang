# v0.9.0 网络稳定性试玩与加固报告

## 1. 问题背景

本轮针对在线长沙麻将的小规模局域网试玩稳定性做发布前加固。重点不是重写麻将规则或调整 UI，而是降低真实联机中常见的重复点击、旧状态提交、断线重连、AI 托管切换和服务端状态漂移风险。

## 2. 本轮目标

- 审计现有 Socket.IO 在线对局链路中的稳定性机制。
- 在不改动核心麻将规则的前提下，补齐小范围工程防护。
- 增加可回归的自动化测试，覆盖重复动作、过期动作、状态版本同步、断线重连、AI 托管和状态不变量审计。
- 生成一份可用于 v0.9.0 试玩的检查报告。

## 3. 当前网络链路审计发现

已有能力：

- `actionId` 已有基础去重缓存，能阻止同一客户端动作被多次执行。
- 等待区和对局中重连已有 `sessionId` / `token` 校验。
- 对局中断线不会立即释放座位，超过阈值后进入托管或离线保留。
- 服务端可按座位生成可见视图，已有隐藏信息隔离测试。
- 网络动作入口已有座位、阶段、轮次和可用动作校验。

本轮补齐点：

- 缺少客户端与服务端之间的 `stateVersion` / `lastEventId` 同步字段。
- 服务端对旧状态提交缺少明确拒绝码。
- 客户端收到旧广播时没有统一忽略策略。
- duplicate action 返回值缺少结构化标记和统计。
- 服务端状态变化后缺少开发模式下的自动不变量审计 hook。

## 4. actionId / clientActionId 方案

本轮沿用现有 `actionId` 字段，不新增并行的 `clientActionId` 命名，避免两个幂等键同时存在造成歧义。

- 客户端每次出牌或响应动作生成唯一 `actionId`。
- 服务端通过 `isDuplicateAction` / `markActionProcessed` 执行幂等防护。
- 客户端提交动作时附带当前可见视图的 `stateVersion`。
- 服务端在处理动作前比较 `action.stateVersion` 与 `session.stateVersion`。

## 5. 重复动作防护

服务端重复动作处理保持幂等语义：

- 同一 `actionId` 再次提交不会再次改变牌局状态。
- 返回 `success: true`、`ignored: true`、`errorCode: 'DUPLICATE_ACTION'`。
- `GameSession.duplicateActionCount` 记录重复动作计数。
- `GameSession.actionAuditLog` 记录重复动作来源座位、版本号和时间。

新增回归覆盖：

- 重复出牌只进入弃牌区一次。
- 重复暗杠请求只提交一次，AI 步进后只形成一个杠副露和一次补牌。

## 6. 过期动作拒绝

服务端新增旧版本动作拒绝：

- 若客户端提交的 `stateVersion` 与服务端当前 `session.stateVersion` 不一致，直接拒绝。
- 返回 `success: false`、`errorCode: 'STALE_ACTION'`。
- 拒绝时不改手牌、不改弃牌、不推进状态版本。

这能覆盖用户在旧画面上点击、网络延迟导致旧动作晚到、重连前后旧 socket 误操作等风险。

## 7. stateVersion 同步

新增 `state-version-utils.ts` 统一管理版本字段：

- 新局初始化为 `stateVersion = 1`、`lastEventId = 'state-1'`。
- 服务端状态成功变化后调用 `markGameSessionStateChanged` 自增版本。
- `buildPlayerVisibleView` 在每个座位视图中携带 `stateVersion`、`lastEventId`、`serverTime`。
- 服务端广播时把房间当前版本写入每个玩家可见视图。

## 8. 旧广播忽略

客户端 hook 新增 `shouldAcceptIncomingView`：

- 当前无视图时接受服务端视图。
- 任一视图缺少版本号时保守接受，兼容旧数据。
- 新视图版本大于或等于当前版本时接受。
- 新视图版本小于当前版本时忽略，避免旧广播覆盖新状态。

## 9. 断线重连方案

本轮复核并保留现有重连机制：

- `game:sync` 继续校验 `roomId`、`seat`、`sessionId` 和 `token`。
- 新 socket 通过校验后恢复座位视图。
- 旧 socket 在重连后不能继续代表该座位行动。
- 等待区断线超时可释放座位，对局中断线保留座位并进入重连/托管流程。

## 10. AI 托管与人类切换

本轮未改 AI 决策和麻将规则，只在 AI 执行后补版本与审计：

- `stepAIIfNeeded` 执行后若牌局状态对象变化，递增 `stateVersion`。
- AI 处理后进入同一个开发模式状态审计 hook。
- 既有测试继续覆盖断线 30 秒后托管、120 秒后离线保留、重连后恢复人类控制。

## 11. 状态审计 hook

新增开发模式审计入口：

- 环境条件：`NODE_ENV !== 'production'` 且 `ENABLE_STATE_AUDIT === '1'`。
- 审计函数：复用 v0.8.9 的 `auditGameStateInvariants`。
- 触发点：玩家动作成功改变状态后、AI 步进后。
- 输出：发现异常时打印 `[StateInvariantViolation]` 和结构化 violations。

## 17. 手工 LAN 试玩结果

本轮在当前终端内未完成真实 2-4 台设备的人工局域网试玩。已完成的替代验证是自动化 Socket/会话级回归，包括重复动作、断线重连、旧 socket 失效、托管恢复、隐藏信息隔离、旧状态拒绝和状态版本同步。

建议人工 LAN 试玩 checklist：

- 2 人建房并补 AI，至少完成发牌、出牌、碰/杠/过、结算中的任意一局流程。
- 同一玩家连续快速点击出牌或杠，确认不会重复打牌或重复杠。
- 一名玩家断网或关闭页面 30 秒以上，确认进入托管。
- 原玩家在 120 秒内重连，确认恢复人类控制且旧页面不能继续操作。
- 在一台设备网络较慢时观察是否出现状态倒退或旧画面覆盖。

## 18. 已知限制

- `stateVersion` 当前按服务端会话状态变化递增，不是完整事件溯源系统。
- duplicate 审计记录仅保存在内存会话中，重启服务后不会保留。
- 客户端旧广播忽略只按版本号判断，不处理跨房间复用视图等更复杂事件流场景。
- 本轮未实现服务端事件重放、断线期间差量补发或持久化房间状态。
- 本轮未做真实多设备 LAN 人工试玩，需要在发布前补一次实际设备验证。

## 19. 下一步建议

- 在局域网人工试玩时保留一份手工记录，补充设备数量、浏览器、断线方式和异常截图。
- 下一阶段可把 `lastEventId` 扩展为真实事件流水号，用于断线后的差量同步。
- 若线上使用频率上升，可把 duplicate/action audit 输出接入开发日志面板或服务端结构化日志。
- 对体积超过 500 kB 的前端 chunk 单独做 code splitting，不与网络稳定性修复混在同一改动中。
