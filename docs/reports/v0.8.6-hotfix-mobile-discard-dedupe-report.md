# 长沙麻将 v0.8.6-hotfix-mobile-discard-dedupe 版本报告

## 一、问题描述

在移动端多人联网对局实测中，玩家打出的普通弃牌（discards）在界面中出现了重复显示。一方面，弃牌已经正确地展示在了中间区域的“各方最近出牌”Dock 以及“中心弃牌河”中；另一方面，同样的普通弃牌也重复展示在对手的玩家信息卡下方（对手面前区域）。这导致玩家难以区分普通弃牌与吃/碰/杠形成的副露（melds），造成视觉语义混乱、信息冗余，严重影响手机端的游戏体验。

---

## 二、用户意图确认

用户明确要求：
1. **移动端普通弃牌只在中间区域显示**（包括 `MobileLatestDiscardDock`、`MobileCenterDiscardArea` 和历史 Drawer）。
2. **玩家面前区域禁止显示普通弃牌**，只能用来显示吃、碰、杠形成的副露。
3. **副露与普通弃牌严格分离**：弃牌归中间牌河，副露归玩家面前。

---

## 三、根因分析

经过审查代码，定位到以下两个关键点：
1. **[MobileOpponentStrip.tsx](file:///c:/Users/hn511/Desktop/VibeCoding/RealMapTeach/src/changsha-mahjong-network/components/MobileOpponentStrip.tsx)** 中的第 60-66 行原本直接渲染了 `player.discards`，在每个对手的面板下方输出了一条横向滚动的弃牌河，造成了和中心弃牌区域的重复渲染。
2. 对手区域虽然也渲染了 `player.melds`（通过 `player.melds.map` 手动渲染 Tile 列表），但未统一格式，也未能在无副露时保持整洁。

## 五、详细修改说明

### 1. 移除玩家面前普通弃牌渲染
移除了 `MobileOpponentStrip.tsx` 中对 `player.discards` 的任何读取与渲染。普通弃牌不再渲染在对手的信息卡下方或面前区域。

### 2. 保留中间牌河与 Dock
中间区域的三个弃牌组件保持不变，未作任何功能上的删减：
* `MobileLatestDiscardDock`：继续显示每个玩家的最新一张出牌。
* `MobileCenterDiscardArea`：继续作为移动端主牌河，展示各家最近的弃牌历史。
* `MobileDiscardHistoryDrawer`：继续支持通过点击中心牌河打开，查看完整的弃牌日志。

### 3. 玩家面前区域只显示副露
在 `MobileOpponentStrip` 中，引入并调用了 `MobileMeldArea`：
```tsx
{/* Only render melds (chi/peng/gang) — discards are shown in center area */}
{player.melds && player.melds.length > 0 && (
  <MobileMeldArea melds={player.melds} compact={true} />
)}
```
只在对手有吃、碰、杠形成的副露时渲染副露区，不填补任何普通弃牌，无副露时该区域为空且不占用多余高度。

### 4. 我的副露回归验证
修改完全在 `MobileOpponentStrip` 中进行，底部“我的手牌”和“我的副露”所使用的 `MobileOnlineGameLayout` 以及 `MobileMeldArea` 未受到任何负面影响。我自己的吃、碰、杠副露依然能正常渲染在底部，我的普通弃牌也不会重复出现在我的副露区。

### 5. `latestDiscardEvent` 与 `actionSourceEvent` 回归验证
最新弃牌的高亮计算、吃碰杠操作的响应来源牌高亮、以及出牌者身份显示（例如 "玩家3 打出 2筒"）功能由 `latest-discard-helper.ts` 驱动，完全保留，高亮定位依然准确。

### 6. activeSeats 兼容说明
本版本没有改动玩家的过滤和人数局映射。对 2人局、3人局、4人局的 activeSeats 识别和渲染依旧保持完美兼容。

### 7. 暗牌隔离复核说明
去重修改仅涉及 UI 渲染的过滤（将本来就公开在牌河的弃牌从对手面板中移除，只在中心牌河显示），不触及 `server-visible-view.ts` 任何数据过滤逻辑。未暴露任何不应公开的对手手牌或暗杠牌信息。暗牌隔离机制 100% 安全。

## 七、已知限制与后续建议

### 1. 已知限制
* 移动端副露区当有过多副露（如 4 组杠牌）时，在某些极小屏幕手机上可能会出现微量的纵向换行，但因为采用了 Flex Wrap，布局不会发生重叠错乱。

### 2. 下一阶段建议
* 可以安全地推进桌面端大字牌渲染优化、或吃碰杠特效提示等视觉增强功能。
