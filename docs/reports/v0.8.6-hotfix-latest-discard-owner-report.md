## 1. 问题描述

在先前的版本中，桌面端和移动端的“全场最新弃牌”在特定情况下存在逻辑与显示偏差：
1.  **识别对象错误**：最新弃牌容易被错误识别并一直显示为“我 打出”的最后一张弃牌，即使其他玩家刚刚打出了新牌。
2.  **出牌者身份缺失**：桌面端 Spotlight 仅显示“最新出牌：9万”，没有标明这张牌是由谁打出的。
3.  **响应源混淆**：当玩家需要对某张牌做出吃、碰、杠、胡响应时，该响应来源牌会直接覆盖“全场最新弃牌”，导致两个概念混淆不清。
4.  **已收纳牌错误高亮**：某张弃牌在被吃、碰、杠收容后，依然作为“牌河中的最新弃牌”错误地在中心河流中保持金色高亮。

---

## 2. 根因分析

1.  **混用 `sourceTile`**：此前在计算 `lastDiscardTile` / `desktopLatestTile` 时，第一优先级直接返回了 `sourceTile`。这导致只要玩家有动作挂起，最新出牌就会被强行覆盖成动作源牌。
2.  **缺乏出牌者追溯**：计算最新出牌时没有将“出牌者的 seat、playerName”同 Tile 一并返回封装成统一事件对象，导致 UI 渲染时无法读取出牌人身份。
3.  **缺少牌河存在性校验**：回退机制及计算没有向前递归核实日志对应的弃牌是否依然还在该出牌者的 `discards` 数组中。如果该牌已经被碰走，由于未进行去重校验，高亮依然会遗留在不复存在的 instanceId 上。

## 4. `latestDiscardEvent` 新计算逻辑

重构为统一的 `getLatestDiscardEvent(view, selfSeat)` 工具函数：
1.  倒序遍历 `view.logs`，定位最近的一条 `action` 含有 `"打出"` 且 `seat` 不为 undefined 的日志。
2.  获取该出牌者的 seat 后，提取其 discard 数组的最后一张卡牌。
3.  比对日志中打出的 tileKey（例如 `wan_9`）是否与该卡牌匹配。
    - **匹配**：说明此卡牌仍未被碰走（还在牌河中），立即作为 `LatestDiscardEvent` 返回，内含 `stillInRiver: true`。
    - **不匹配**：说明该牌已被吃/碰/杠收纳，程序会自动忽略它，继续向前（向更早的日志）扫描，直至找到首个依然存留在河中的有效弃牌。
4.  如果未在 logs 中找到任何存留的弃牌，则返回 `null`，不再盲目 fallback 误标“我最新打出的牌”。

## 5. `actionSourceEvent` 与 `latestDiscardEvent` 的区别

-   **`latestDiscardEvent` (全场最新弃牌事件)**：
    *   **定义**：代表全局时间线上最近一次“打出牌”且目前仍滞留在牌河中的事件。
    *   **作用**：决定桌面端 Spotlight 顶栏、移动端 Dock 高亮以及牌桌中心河流中的金色强高亮。
-   **`actionSourceEvent` (操作响应来源事件)**：
    *   **定义**：代表当前等待我进行动作响应（吃碰杠胡）的那张牌和出牌者。
    *   **作用**：决定操作 candidate 面板上方的“响应来源”卡片。在完成动作或过牌后，该卡片立刻随着 pendingActions 清空而消失，与全场最新弃牌完全剥离。

## 8. 中心牌河高亮修复说明

-   牌河中每一位玩家的出牌有其局部高亮逻辑。而全场最闪耀的金色强高亮（`highlightType="latest"`）仅当该牌的 `instanceId` 与 `latestDiscardEvent.tile.instanceId` 相同且仍在河中时才会生效。
-   若最新打出的牌已经被碰走收容，最新高亮会通过后向扫描自动落到更早打出且仍留存的那张牌上，绝对不会高亮已被收容消失的牌。

## 9. activeSeats 兼容说明

-   所有计算均使用 `view.opponents` 及本地 `seat` 拼合后的 active seats。
-   如在 2 人局中，若 inactive 玩家 3（Seat 3）的占位日志被解析出来，但由于其 seat 不在 activeSeats 中，工具函数将跳过它，防止在 UI 上渲染出“玩家3”等幽灵座位提示。

## 10. 吃碰杠后弃牌收纳场景说明

-   当玩家打出 `wan_3`，日志记下 `打出 wan_3`。此时该牌是最新弃牌。
-   随后其他玩家选择碰牌（收纳 `wan_3`），该牌被从原本的 discard 河流移入副露。
-   下一次重绘时，`getLatestDiscardEvent` 扫描到 `打出 wan_3` 日志，但发现出牌者的河流末尾已非 `wan_3`，于是继续向下追溯，找到更早的一条打出日志（例如 `玩家0 打出 tong_7`），并将高亮成功转移到 `tong_7` 上，表现完全符合物理直觉。
